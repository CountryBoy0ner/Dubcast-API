# Documentation Strategy for Dubcast Radio API

## 1. Purpose of Documentation

This documentation describes all external interfaces of Dubcast Radio
API.\
Its goal is to provide developers with a clear, consistent, validated,
reproducible API reference, ensuring that the system can be easily
integrated, maintained, and extended.

The documentation covers: - API reference - Guidelines for using the
API - Error format - Examples of requests and responses - High-level
architecture and context of the API - Versioning rules

------------------------------------------------------------------------

## 2. Documentation Tools Used

### 2.1 OpenAPI / Swagger Specification

The primary API specification format is **OpenAPI 3.1**.

Tools used: - **Springdoc OpenAPI** -- automatic generation from
annotations - **Swagger UI** -- interactive view - **openapi.json** --
autogenerated and versioned - **openapi.yaml** -- manually
exported/converted version for readability - **Swagger Editor** --
validation, visual verification

### 2.2 Additional Tools

-   **Spectral (Redocly)** -- optional linter for best practices
-   **cURL examples** -- manually prepared guides
-   **Markdown docs** -- GitHub-friendly documentation

------------------------------------------------------------------------

This structure follows industry standards: - `schemas` →
machine-readable specifications\
- `reference` → normative descriptions\
- `guides` → tutorials\
- `examples` → ready-to-use commands

------------------------------------------------------------------------

## 4. Naming Conventions

### 4.1 Endpoints

-   Lowercase\
-   Hyphen-separated when needed\
-   Plural nouns for collections\
-   Format: `/api/{resource}`

Example:\
`/api/admin/programming/day/{date}/insert-track`

### 4.2 Data Models (DTOs)

-   PascalCase for DTO names\
-   camelCase for JSON fields

------------------------------------------------------------------------

## 5. Versioning Strategy

Current version: **v1**

Rules: - Breaking changes → require `/api/v2/*` - Non-breaking additions
allowed in v1 - OpenAPI `info.version` reflects public API version

------------------------------------------------------------------------

## 6. Formatting Rules

### JSON examples:

-   2 spaces indentation\
-   no trailing commas

### Markdown:

-   triple backticks for code blocks\
-   logical header levels\
-   examples in `/docs/examples`\
-   guides in `/docs/guides`

------------------------------------------------------------------------

## 7. Known Limitations

-   Some admin endpoints may evolve as UI grows\
-   Future modules (websockets, streaming) may require new endpoints

------------------------------------------------------------------------

## 8. Validation of Specification

Validated using: - Swagger Editor\
- Spectral\
- Springdoc

Ensures correct schema, formats, types, and references.

------------------------------------------------------------------------

## 9. Summary

The documentation strategy ensures: - consistent API description\
- validated and versioned specification\
- clear guides and examples\
- maintainability and extensibility

This satisfies the minimum diploma requirements for API documentation.
